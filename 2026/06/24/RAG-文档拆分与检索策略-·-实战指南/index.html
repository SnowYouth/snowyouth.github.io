前言:为什么这两步决定 RAG 的天花板

很多人第一次跑 RAG 失败,会去换 embedding 模型、换向量库、改 prompt,但通常真正的瓶颈在两个早期决策上:

  1. 切块方式不对 —— 答案散落在多个块里,或者被切到中间断掉。
  2. 检索方式过于朴素 —— 只用向量 top-k,该召回的内容根本没排上前几名。

后续无论换什么大模型,也回不到这两个环节丢掉的信息。

形象点说:RAG 是一个三段管道——

第三段(模型生成)是最常被关注的,但前两段才决定上限。本文专攻前两段。

第一部分:文档拆分策略 (Chunking)

1. 为什么必须切块

最直接的原因是上下文窗口有限。 即使现在的模型支持 128K、200K 甚至 1M token,把整本几百页的文档塞进去仍然不现实:

  • 成本:输入 token 是按量计费的,塞 100K 上下文意味着每次问答都是一笔不小的开销
  • 延迟:更长的上下文 = 更慢的首 token 响应
  • 注意力稀释:模型对长上下文的检索能力 会随长度显著下降(著名的 “Lost in the Middle” 现象——开头和结尾记得最清楚,中间往往被忽略)
  • 信噪比:与问题无关的内容会干扰生成

第二个原因是检索粒度。 要想准确召回 “用户问的那段话”,索引单元必须比”整篇文档”更细。但又不能细到一句话——孤立一句话往往脱离上下文不可理解。

所以切块的本质是:找到一个能保持语义自足、又足够细粒度可被检索定位的最小单位。

什么是”语义自足”?一个块单独看出来,人读得懂,知道它在讲什么。如果一个块开头是”这个方法的优点是…”,但前文没出现”这个方法”指的是什么,这块就没有自足语义。

2. 主流切块策略全景

下文每种策略都按统一格式介绍:原理 → 示例 → 优势 → 劣势 → 适用场景。

2.1 固定长度切块 (Fixed-size)

原理: 按固定字符数(或 token 数)硬切。比如每 500 个字符一段。

优势:

  • 实现极其简单,几行代码
  • 完全可预测——给一段 N 字符的文本,你能预先算出会切成几块
  • 适合做基准对比 (baseline)

劣势:

  • 位置敏感:在文章开头插入 10 个字,后面所有块的边界全部漂移,所有块的内容、hash 全部变化。这对增量更新极其不友好
  • 粗暴跨语义边界:一句话、一个表格、一段代码经常被切成两半
  • 完全无视文档结构——标题、列表、代码块统统不认

适用场景:

  • 纯实验/做对照组
  • 文本本身没有任何结构(比如 OCR 出来的乱码、聊天记录、日志流)
  • 你只关心召回率不关心精度的快速原型

结论: 几乎不该作为生产方案。它是 RAG 教程里最常出现、但实战中最快被淘汰的策略。

2.2 重叠切块 (Sliding Window / Overlap)

原理: 在固定长度切块的基础上,让相邻两块共享一段重叠区域 (比如重叠 50-100 字符)。这样原本被切断的语义有机会同时出现在两个块里。

优势:

  • 缓解了固定切块的”语义被腰斩”问题
  • 实现仍然非常简单
  • 重叠区也提供了上下文锚点,块的可读性提升

劣势:

  • 仍然位置敏感 (跟固定切块一个问题)
  • 重叠意味着存储与索引膨胀,典型情况下索引体积会增加 20%~40%
  • 检索时同一段话可能被多个块同时召回,top-k 里出现冗余,挤占了别的相关内容的位置(需要后处理去重)
  • 重叠多少全靠拍脑袋

适用场景:

  • 作为固定切块的快速升级——成本极低,收益明显
  • 文档无明显结构,但希望保住一定的跨边界召回率

小贴士: 重叠通常设为 chunk_size 的 10%-20%。低于 10% 几乎没用,高于 30% 性价比急剧下降。

2.3 句子级切块 (Sentence-based)

原理: 用句末标点(中文: 。!?;,英文: . ! ?)做切分。每个块是 1 个或 N 个完整句子。

优势:

  • 每个块都是完整句子,可读性好
  • 不再有”切到一半”问题
  • 边界跟随语义,对增量更新友好——改一句话只影响这一句所在的块

劣势:

  • 句子粒度对长文档而言太细——一篇 1 万字的文章可能切成几百块,检索压力大
  • 单句往往缺乏自足语义(代词、指代依赖前后文)
  • 中文句号识别要小心(省略号、引号内的句号、数字小数点)

适用场景:

  • 短文档:FAQ、问答对、字幕、微博/推文
  • 法律条款、合同条文——每条款本身就是检索单位
  • 作为后续”按 N 句聚合成块”的预处理步骤

2.4 段落级切块 (Paragraph-based)

原理: 用空行 (\n\n) 或连续换行做边界,一段为一块。

优势:

  • 段落是作者自己划分的语义单位,切出来的块最贴近人类阅读直觉
  • 块大小自适应——短段落自然小,长段落自然大
  • 跟句子级一样对增量友好

劣势:

  • 段落长度差异极大,有的两行,有的两屏。短段不够上下文,长段超出 embedding 模型输入长度
  • 强依赖原文排版——很多 PDF/Word 提取出来根本没有正确的段落标记
  • 同一概念可能分散在多段(比如”第一,…第二,…第三,…”)

适用场景:

  • 排版良好的 Markdown、博客、技术文档
  • 作为递归切块的第一层依据
  • 多数 Markdown 笔记类知识库

2.5 递归切块 (Recursive)

原理: 给定一组优先级递降的分隔符 (例如 ["\n\n", "\n", "。", " ", ""]),按顺序尝试切分:

  1. 先按 \n\n (段落) 切,如果某块还超过 max_size,就在这块内
  2. 按 \n (行) 继续切,还超就
  3. 按句号切,直到
  4. 实在没办法就按字符硬切

优势:

  • 集众家之长:优先保住段落、其次句子、最后才硬切
  • 几乎所有切块库的默认推荐(LangChain 的 RecursiveCharacterTextSplitter、LlamaIndex 默认行为都是这个思路)
  • 实现简单、性能高、效果稳健

劣势:

  • 分隔符列表需要按语种调整(中英标点要分开列)
  • 仍然不感知文档结构——它不知道 # 标题 和正文的语义层级不同
  • 对代码、表格仍然会乱切

适用场景:

  • 绝大多数纯文本场景的合理默认
  • 文档结构不规则但内容是自然语言为主

结论: 不知道选什么的时候,从递归切块开始。基本能拿到 75 分以上。

2.6 结构感知切块 (Structure-aware: Markdown / HTML / Code)

原理: 利用文档自带的结构标记切块。

  • Markdown:按 #、##、### 标题层次切,每个章节(含其下所有内容)是一个块,或者每个 ## 是一个块
  • HTML:按 <h1>、<h2>、<article>、<section> 等语义标签切
  • 代码:按函数、类、方法切(借助 AST 或 tree-sitter 这类语法解析器)

优势:

  • 跟着作者写作时的语义单位走——作者把这一节单独立标题就是在告诉你”这是一个独立话题”
  • 边界稳定:在某一节内增加内容,只影响该节这一个块,其他块完全不变
  • 可以同时把标题路径作为 metadata 存下来(比如 “技术文档 > 安装 > Windows 平台”),既可用于过滤,也可在生成时作为上下文增强

劣势:

  • 强依赖文档格式——纯文本、扫描 PDF 用不了
  • 章节长度差异可能很大,需要再叠一层”超大块再细分”的兜底逻辑
  • 代码切块需要每种语言一个解析器

适用场景:

  • 技术文档、API 文档、README、博客
  • 代码搜索/代码 RAG (Github Copilot 类场景)
  • 法规、教材、论文这种结构强的文本

实操建议: 结构感知 + 递归兜底 (块太大时退化为递归切) 是当前文档型知识库的最佳实践之一。

2.7 语义切块 (Semantic)

原理: 把文档先拆成句子,对每句做 embedding,然后顺着读——当相邻两句的 embedding 相似度突然下降(超过设定阈值),就在这里切一刀。直觉上是:话题变了,该开新块了。

优势:

  • 边界基于语义,理论上最贴合”自然话题切换”
  • 不依赖任何格式标记,纯文本也能用
  • 块大小完全自适应内容

劣势:

  • 计算成本高——每个文档要先做一遍逐句 embedding 才能切
  • 阈值难调,数据集换一批可能就要重调
  • 对写作风格敏感:有的人写作话题切换不明显,阈值灵敏度差;长篇大论的散文型文本切不开
  • 一次 embedding 模型升级,所有边界都要重算

适用场景:

  • 长篇无结构文本:小说、访谈实录、会议记录、长文章
  • 你愿意付出预处理成本换取边界质量的高价值知识库
  • 已经有了 embedding pipeline,边际成本不高的项目

2.8 父子切块 (Parent-Child / Small-to-Big)

原理: 同一份文档索引两份——

  • 子块 (child):细粒度 (比如 200 字),用于 embedding 和检索匹配
  • 父块 (parent):粗粒度 (比如 2000 字,或整章),用于送给 LLM 生成

检索时:用 query 匹配子块 → 拿到子块对应的父块 → 把父块送进生成。

优势:

  • 检索精度高 (用小块匹配,关键词命中精准)
  • 生成上下文丰富 (用大块送给 LLM,模型能看到完整背景)
  • 解决了 “小块召得准但上下文不够 / 大块上下文够但召不准” 的两难

劣势:

  • 索引结构变复杂——要维护父子映射关系
  • 存储约翻倍 (子块向量 + 父块原文)
  • 多个子块可能映射到同一父块,top-k 召回时容易出现父块重复 (需要去重逻辑)
  • 实现/调试成本明显高于普通切块

适用场景:

  • 长文档 (论文、书籍、报告) RAG
  • 答案需要丰富上下文 (技术问答、案例分析)
  • 已经把 baseline RAG 跑稳、要进阶提分的项目

2.9 命题切块 (Proposition-based)

原理: 用 LLM 把文档重写成一条条独立的、自包含的事实陈述 (proposition)。每条陈述是一个块。

例子:

原文: “李白是唐代著名诗人,他的代表作《将进酒》气势磅礴。”

命题化后:

  • 李白是唐代诗人。
  • 李白是著名诗人。
  • 《将进酒》是李白的代表作。
  • 《将进酒》气势磅礴。

优势:

  • 每条命题都完全自足,没有指代依赖,不存在”这个/那个/它”
  • 检索精度极高——每条命题就是一条原子事实
  • 非常适合事实型问答 (factoid QA)

劣势:

  • 预处理成本极高——每篇文档要走一遍 LLM 重写,贵且慢
  • 重写质量取决于 LLM,可能引入幻觉或丢信息
  • 命题量爆炸——一段话可能拆成几十条命题,索引体积膨胀
  • 重写后失去原文档的叙述结构,生成时上下文可能显得碎片化

适用场景:

  • 高价值、相对静态的知识库 (产品手册、医学文献)
  • 强事实型场景 (法律条款查询、医学问诊辅助)
  • 一次性投入大量预处理换长期检索精度的项目

2.10 Late Chunking

原理: 一种 2024 年提出的较新思路。常规做法是”先切块、再各自 embedding”;Late Chunking 反过来——先把整段(或整篇)文档喂给长上下文 embedding 模型,在最后一层得到每个 token 的向量后再做切分聚合,得到每个块的向量。

直觉是:每个 token 的 embedding 在模型生成时已经”看过”了整篇上下文,所以即使切成小块,每个块的向量也”带着全局上下文”。这缓解了小块脱离上下文的问题。

优势:

  • 块向量天然带有跨块上下文,小块也能保留全局语义
  • 切块策略可以更激进 (更小、更细),不用担心丢上下文
  • 不需要预先做命题改写

劣势:

  • 需要支持长上下文的 embedding 模型 (如 jina-embeddings-v3)
  • 实现新颖,工具链支持还不普及,大多数现成框架不直接提供
  • 单文档 embedding 计算开销大

适用场景:

  • 探索性项目、研究型 RAG
  • 有富余 GPU 资源且追求 SOTA 效果的场景
  • 不愿意为父子结构付出额外存储成本但又想要”块小且带上下文”的项目

2.11 Agentic Chunking

原理: 不写死任何规则,让一个 LLM Agent 自己读文档、自己决定该怎么切。Agent 可以基于内容类型 (代码 / 表格 / 散文)、章节边界、话题密度等综合判断。

优势:

  • 灵活性极高,理论上每种文档类型都能找到最优切法
  • 可以处理混合内容 (一篇文档里既有正文又有代码又有表格)

劣势:

  • 成本最高——切块本身就要花 LLM 调用
  • 不可复现——同一份文档切两次可能得到不同结果
  • 难以调试——切坏了不知道是 prompt 问题还是模型问题
  • 大规模场景下基本无法承受成本

适用场景:

  • 小规模高价值文档
  • 研究、demo
  • 几乎不推荐用于生产 RAG (除非有专门的优化)

3. 通用调参:chunk_size 与 overlap 怎么定

不管用哪种策略,几乎都绕不开这两个参数。

chunk_size

经验范围:

场景 建议字符数 (中文) 建议 token 数 (英文)
短问答、FAQ 100~300 64~128
一般文档检索 300~800 256~512
长文档、需要丰富上下文 800~1500 512~1024
父子切块的父块 1500~3000 1024~2048

怎么定 (从两端往中间想):

  • 上限取决于:embedding 模型输入长度、生成模型上下文预算、Lost in the Middle 容忍度
  • 下限取决于:一个块能不能讲完一件事 (语义自足)

有个简单测法: 拿真实问题去问自己——“一个块大概要有多少内容,模型读了它就能正确回答这个问题?” 这个量就是合理的 chunk_size。

overlap

经验值: chunk_size 的 10%-20%。

什么时候加大:

  • 文档中存在大量跨边界引用 (代词、”如上所述”)
  • chunk_size 偏小

什么时候不需要 overlap:

  • 用了语义切块、结构感知切块 ——边界本身就在自然停顿点
  • 检索后有 rerank 兜底
  • 索引存储是瓶颈

本质: overlap 是对”切块不够聪明”的弥补。切得越聪明,需要的 overlap 越少。

5. 切块常见坑

坑 1:用 \n 当段落分隔符。
很多文档段落用 \n\n 而不是 \n,后者只是行内换行。混用会切碎正文。

坑 2:embedding 截断没考虑。
embedding 模型有最大输入长度 (常见 512 token)。如果块超过这个长度,要么被截断 (后半段丢失),要么报错。chunk_size 应略小于模型上限。

坑 3:中文按 token 切等于按字节切。
英文 1 token ≈ 4 字符,中文 1 token ≈ 1-2 字符。同样 512 token,英文能装 2000 字符,中文只有 600-800 字符。给中文配 chunk_size 时不要照搬英文教程数字。

坑 4:增量更新没考虑边界稳定性。
固定/重叠切块在小改动后会让大量块的 hash 漂移,导致整片重新 embedding (很贵)。结构感知/句子/段落切块天然边界稳定,改一节只影响一节。

坑 5:把元数据(标题路径、来源、修改时间)扔了。
切完块只存正文不存来源,后续既不能做过滤、也不能给用户展示引用,等于自废一半武功。

第二部分:检索策略 (Retrieval)

1. 为什么”向量检索 top-k”远远不够

教程里的标准 RAG:把文档切块、做 embedding、扔向量库、用户问题也 embedding、取最近的 k 个块。

这个流程能跑,但效果天花板很低,原因有四:

(a) Query-Document Gap. 用户问”怎么登录?”,文档里写的是”账号验证流程”。两段话语义相关、字面无关,纯关键词检索抓不到,纯语义检索分数也偏低。

(b) 关键词命中场景被忽视。 用户问”DEADBEEF 错误怎么解决?”——“DEADBEEF” 是一个特定错误码,向量模型对低频专有名词的表征往往很弱,反而是老掉牙的 BM25 一击即中。

(c) 排序粒度太粗。 向量相似度只算两个向量的几何距离,没有真正”理解” query 和 doc 之间的交互。所以即使 top-30 里有正确答案,它也未必排在前 5。

(d) 单一 query 视角太窄。 用户提问短而模糊,一次检索覆盖不到所有相关概念。同一个意图可以有几十种问法。

对应解决方案就是本节后面要讲的所有内容。

2. 主流检索策略全景

2.1 稠密向量检索 (Dense / Vector)

原理: 把 query 和文档块都用 embedding 模型转成向量,在向量空间里找最近邻 (常用 cosine 相似度)。

优势:

  • 强语义召回——同义词、近义词、改写都能找到
  • 跨语言能力强 (多语言模型可以中文 query 搜英文文档)
  • 实现成熟、向量库 (Chroma/Milvus/Qdrant/PGVector) 选择多
  • 跟切块、embedding 模型解耦,可独立升级

劣势:

  • 对罕见专有名词、术语、ID、错误码召回弱 (这些低频词的向量表征质量差)
  • 完全字符匹配的需求 (找”包含 ‘X’ 的所有文档”) 表现不佳
  • 召回质量强依赖 embedding 模型,模型一换,所有索引重建
  • “排得高 ≠ 真相关”,几何距离不是真正的语义判断

适用场景:

  • 自然语言问答场景
  • 跨语言检索
  • 几乎所有 RAG 项目的最基本检索层

2.2 稀疏检索 (Sparse: BM25 / TF-IDF)

原理: 经典信息检索算法。把 query 和文档拆成词,基于”该词在 query 出现 + 在该文档出现 + 在整个语料库的罕见程度”算分。BM25 是 TF-IDF 的改良版,加入了文档长度归一化和饱和函数。

优势:

  • 关键词命中精准:专有名词、术语、错误码、人名、API 名一击即中
  • 不需要任何模型,纯统计算法——冷启动零成本
  • 结果可解释——能直接看出”这篇是因为命中了 ‘XXX’ 才排前面”
  • 速度极快,内存友好
  • 多年工业级实践,Lucene / Elasticsearch / Tantivy 都内置

劣势:

  • 没有语义 ——“登录” 和 “签到” 在 BM25 眼里完全无关
  • 同义词、改写、跨语言全部失效
  • 中文必须先做分词 (jieba / hanlp / Tantivy 内置中文分词);分词错则检索失效
  • 对短 query 和长文档不太敏感 (虽然 BM25 已经做了归一化但仍有偏差)

适用场景:

  • 技术文档、API 文档、错误码查询
  • 用户习惯输入精确关键词的场景 (搜索框风格)
  • 作为混合检索的一部分 (几乎必备)

原理: 同时跑向量检索和稀疏检索,把两个结果列表融合成一个最终排名。两种常见融合方式:

优势:

  • 互补效应明显:语义召回 + 关键词命中,一份顶两份
  • BM25 兜住了向量模型对罕见词的弱点
  • 向量检索兜住了 BM25 的同义词盲区
  • RRF 实现简单 (10 行代码),效果稳定

劣势:

  • 需要双索引,存储、增量更新、运维成本翻倍
  • 中文 BM25 需要分词预处理 (引入 jieba 等依赖)
  • 调参变多 (alpha、k、各自 top-k)
  • 两个索引可能不同步 (一边加了文档另一边漏了)

适用场景:

  • 几乎所有生产 RAG 的默认配置
  • 包含大量专有名词、术语、代码、配置项的知识库
  • 用户 query 既有自然语言也有关键词混合

结论: 如果只能给基础 RAG 加一个改进,加 Hybrid。

2.4 重排序 (Reranking / Cross-encoder)

原理: 第一阶段(粗排)用快速检索器 (向量/BM25/Hybrid) 召回 top-30 或 top-50;第二阶段(精排)把 query 和每个候选块拼起来一起喂给一个 cross-encoder 模型,直接输出一个相关性分数,按这个分数重新排序,取最终 top-5。

关键区别:

  • 普通向量模型 (bi-encoder): query → vec_q,doc → vec_d,然后算几何距离。query 和 doc 互相不知道对方。
  • Cross-encoder: 把 [query, doc] 一起塞进模型,模型的 attention 可以让 query 和 doc 的每个 token 互相关注。精度高一档。

为什么不一开始就用 cross-encoder? 因为成本是 O(query × N_docs) 而不是 O(query + N_docs)——你不可能对全库每个块都跑一遍 cross-encoder,只能用于粗排后的 top-N。

优势:

  • 精度显著提升:行业经验,Hybrid + Rerank 通常比纯 Hybrid 再涨 10%-20% 的相关性指标
  • 对 query-doc 表述差异容忍度高
  • 实现简单 —— 接入一个 reranker 模型即可

劣势:

  • 延迟增加:每次查询多 30-50 次模型 forward,CPU 上 1-3 秒,GPU 上 100-300ms
  • 引入了额外模型 (典型 reranker 大小 300MB-1GB)
  • 对小知识库收益小——如果 top-5 已经包含正确答案,重排没用
  • 候选集太大 (>50) 时延迟会拖垮整体响应

主流 Reranker 模型:

  • 开源:bge-reranker-base / large / v2-m3 (中英都好)、jina-reranker-v2、mxbai-rerank
  • 闭源 API:Cohere Rerank、Voyage Rerank

适用场景:

  • 中大型知识库 (千块以上)
  • 对精度敏感的场景 (法律、医学、技术支持)
  • 已经把 Hybrid 跑稳,想再提一档

2.5 多查询检索 (Multi-Query)

原理: 用 LLM 把原始 query 改写出 3-5 个不同表达的变体,每个变体各检索一次,把结果合并去重。

例子:

  • 原 query: “怎么让模型回答更准?”
  • 变体 1: “提升 LLM 回答准确性的方法有哪些?”
  • 变体 2: “如何减少大语言模型的幻觉?”
  • 变体 3: “RAG 中怎么优化生成质量?”

优势:

  • 弥补了用户单次提问视角窄的问题
  • 实现简单 (一个改写 prompt + 多次检索 + 去重)
  • 对抽象、短、模糊的 query 特别有效

劣势:

  • 每次问答多一次 LLM 调用,延迟和成本上升
  • 改写质量取决于 LLM——改差了反而召回更差
  • 变体过多 (>5) 时,合并去重后结果可能仍然冗余
  • 不解决”知识库里压根没这块内容”的根本问题

适用场景:

  • 用户 query 偏短偏抽象 (搜索框 / 对话式)
  • 容忍延迟换召回率的场景
  • Agent 自动生成 query 的场景

2.6 HyDE (Hypothetical Document Embeddings)

原理: 让 LLM 先”假装回答”——基于 query 编一段”如果有答案,大概长这样”的虚构文档,然后用这段虚构文档的向量去检索,而不是 query 的向量。

直觉: “问题”和”答案”在向量空间分布往往不重叠 (“如何登录?” 和 “登录步骤如下…”的语义结构不一样)。假答案的向量更接近真答案。

优势:

  • 对”问题表述和答案表述风格差异大”的场景有效
  • 在某些 zero-shot 检索基准上确实涨点

劣势:

  • 每次都多一次 LLM 调用
  • LLM 编的假答案如果方向偏了,检索反而走偏 (引入新的”幻觉式 query drift”)
  • 在有标注数据可微调 embedding 时,收益基本被 Hybrid + Rerank 覆盖
  • 实战中越来越少单独使用

适用场景:

  • 学术实验
  • query 和 doc 风格反差很大的特定领域
  • 生产中通常被替代,不作为首选

2.7 Step-back Prompting

原理: 跟 Multi-Query 类似,但思路不同——让 LLM 把 query 抽象化、退一步,得到一个更宽泛的 query,然后用宽泛 query 检索 (或者宽泛 + 原始一起检索)。

例子:

  • 原 query: “Python 3.12 里 type hint 怎么写泛型?”
  • Step-back: “Python 类型注解的演进”

优势:

  • 对具体问题需要宽泛背景的场景有效
  • 检索到的内容更利于 LLM 推理

劣势:

  • 同样多一次 LLM 调用
  • 抽象后的 query 可能召回大量不相关泛文,反而稀释精度
  • 收益场景较窄

适用场景:

  • 教学问答 (要给原理背景)
  • 多跳推理任务

2.8 RAG-Fusion

原理: 可以理解为 Multi-Query + RRF。先用 LLM 生成多个 query 变体,每个变体独立检索,然后用 RRF 把多份结果列表融合成一份最终排名。

优势:

  • 比单纯 Multi-Query 多了一个严谨的融合步骤,结果更稳
  • RRF 自带去重和加权,出来的列表质量高

劣势:

  • 综合了 Multi-Query 和融合两步,延迟和成本是基础 RAG 的 3-5 倍
  • 实现复杂度上升

适用场景:

  • 离线/异步 RAG (容忍延迟)
  • 对召回质量极致追求的场景

2.9 元数据过滤 (Metadata Filtering)

原理: 切块时把额外信息 (来源文件、章节路径、作者、日期、文档类型、语言) 作为 metadata 一起存入向量库。检索时先按 metadata 过滤,再做向量/关键词匹配。

例子:

优势:

  • 大幅缩小搜索空间——只在符合条件的子集里检索,精度暴涨
  • 解决了”全库语义搜索找出无关时间/部门内容”的常见问题
  • 实现成本低,几乎所有向量库都原生支持

劣势:

  • 强依赖 metadata 字段设计——字段没设好,过滤没用
  • query 中的元信息需要被识别 (年份、地区、文档类型),否则无法转换为 filter——这一步通常需要 LLM 辅助 (见 Self-Query)
  • 过滤过紧会漏掉相关内容

适用场景:

  • 文档天然有维度划分 (按时间/部门/产品线/语言)
  • 用户 query 自带筛选条件 (含时间、地点、人物的问题)
  • 几乎所有生产 RAG 都应该用

2.10 Self-Query

原理: 用 LLM 自动从 query 中抽取过滤条件,生成结构化的 metadata filter,再去做检索。是 Metadata Filtering 的自动化版本。

例子:

  • 原 query: “2024 年的法律相关博客有哪些讨论了 AI 监管?”
  • LLM 抽取出:
    • filter: {year: 2024, category: "law"}
    • 重写后的语义 query: “AI 监管”

优势:

  • 用户无需手动设 filter,自然语言 query 直接生效
  • 配合 Metadata Filtering 威力放大
  • 减少召回噪声

劣势:

  • 多一次 LLM 调用 (延迟、成本)
  • LLM 抽取错了会过度过滤
  • 需要给 LLM 提供完整的字段 schema,prompt 工程量大

适用场景:

  • 文档库有清晰的 metadata schema
  • 用户 query 经常包含筛选条件
  • 高级 RAG / Agent 场景

2.11 上下文压缩 (Contextual Compression)

原理: 检索到 top-k 块后,在送进 LLM 之前用一个小模型 (或 LLM) 再过滤一遍,只留下真正与 query 相关的句子。直觉是:即使 top-5 块都召得对,每块里也只有 30% 的内容是真正答这个问题的,剩下 70% 是噪声。

优势:

  • 大幅减少送进生成 LLM 的 token,省钱省时
  • 缓解 Lost in the Middle
  • 提升生成答案的精度 (噪声变少)

劣势:

  • 多一次过滤模型调用
  • 压缩过狠会丢上下文
  • 实现稍复杂

适用场景:

  • 召回块比较长 (>500 字符)
  • 对生成成本敏感
  • top-k 已经召得不错,想”提纯”

2.12 图检索 (Graph RAG)

原理: 不只靠向量/关键词召回,还把文档中的实体、关系抽取出来构建知识图谱。检索时一并用图遍历得到多跳关联的信息。比如问”A 公司的 CEO 写过哪些书?”,先找到 “A 公司”→”CEO”→具体人物→其作品。

优势:

  • 能回答多跳推理问题 (纯 RAG 几乎无解)
  • 实体级精准,适合做关系型问答
  • 结构化知识可以做严格事实校验

劣势:

  • 构建成本极高:实体/关系抽取要么写规则要么调 LLM,且质量参差
  • 图的更新和维护是大工程
  • 不是所有场景都用得上 (常规问答 90% 不需要图)
  • 不成熟的 Graph RAG 反而比 baseline RAG 差

适用场景:

  • 强关系型领域:学术研究、企业 CRM、医学知识、金融关联分析
  • 已经有现成知识图谱可用
  • 多跳问答是核心需求

重要提醒: Microsoft 的 GraphRAG、Neo4j 的 LLM Graph Builder 让这条路门槛大幅降低,但别盲目跟风。先把 Hybrid + Rerank 跑稳再来想图。

3. 怎么组合:典型架构与边界

把上面这些策略按”召回阶段”和”精排阶段”分一下,会清楚很多:

三档典型组合

入门级:

  • 向量检索 top-5 → 直接送 LLM

中级 (生产可用底线):

  • Hybrid (向量 + BM25 + RRF) top-30 → Cross-encoder Rerank → top-5 → 送 LLM
  • 配合 Metadata Filtering

进阶 (复杂场景):

  • Self-Query 抽 filter
  • → Multi-Query 改写 3 份
  • → 每份做 Hybrid 召回 top-30
  • → 合并去重
  • → Rerank 取 top-10
  • → Contextual Compression
  • → 送 LLM

选择原则

  1. 从中级方案起步,不要从入门级直接跳进阶。 进阶方案延迟和成本上升明显,没把 baseline 跑稳就上等于在浑水里调参。
  2. Hybrid 是性价比之王,几乎没有不上的理由。
  3. Reranker 在数据量超过几百块后收益开始凸显,几十块的小库可能感觉不到。
  4. Multi-Query / HyDE / Step-back 慎重——它们都是”再 LLM 调一次”换召回扩展,每次 query 多一倍延迟和成本,要看场景是否值得。
  5. Metadata Filtering 是免费午餐,只要你的文档天然有维度划分,从一开始就加上。
  6. 图检索是另一个世界,适合特殊场景,不要把它当 RAG 升级路径上的”必修课”。

4. 检索选型决策树

5. 评估:别凭感觉说”效果不错”

很多团队改了一堆策略,然后说”感觉好多了”,但没有量化。这一节给一个最小可用的评估框架。

准备评测集

收集 50-100 条”问题 + 期望相关文档/期望答案”。来源:

  • 内部历史用户问答
  • 让领域专家手写
  • 用 LLM 基于知识库生成,人审一遍

关键指标

指标 含义 衡量哪一段
Recall@k top-k 里至少有 1 个相关文档的比例 召回阶段
Precision@k top-k 里相关文档的比例 排序质量
MRR (Mean Reciprocal Rank) 第一个相关结果排名的倒数的均值 排序质量
nDCG@k 考虑相关度等级的排名指标 排序质量
Faithfulness 生成答案有没有跑出召回内容之外 生成质量
Answer Correctness 生成答案对不对 端到端

实务上只看三件事就够用:Recall@10、MRR、端到端答对率。

工具推荐

  • RAGAS (开源):自动算 Faithfulness、Answer Relevancy、Context Precision 等
  • TruLens:可视化检索链路
  • LlamaIndex Eval / LangChain Eval:框架自带评测模块
  • DeepEval:类似 RAGAS,工具集成更多

一个最小评测流程

  1. 写 50 条 query + 标注期望相关文档
  2. 跑 baseline (纯向量检索),记录 Recall@10 和 MRR
  3. 每加一个策略 (Hybrid → Rerank → Multi-Query),再跑一次
  4. 看哪些策略真涨,哪些不涨,哪些反而掉
  5. 留下涨的,把不涨的回滚

没有这一步,所有”策略升级”都是猜。

附录 A:术语小词典

  • chunk / 块:文档被切分后用于检索和向量化的最小单位
  • embedding:把文本转成定长向量的过程,或者得到的那个向量
  • bi-encoder:对 query 和 doc 各自独立编码的模型 (普通 embedding 模型属于这类)
  • cross-encoder:把 query 和 doc 拼一起一起编码的模型,精度高但慢 (reranker 属于这类)
  • 稠密向量 (dense):维度固定 (通常几百到几千)、值多为非零的向量,捕捉语义
  • 稀疏向量 (sparse):维度极高 (词表大小) 但大多数为零的向量,典型代表 TF-IDF/BM25
  • BM25:经典关键词检索算法,Lucene/Elasticsearch 等系统的默认评分函数
  • RRF (Reciprocal Rank Fusion):基于排名的多路检索结果融合方法
  • top-k:检索时返回相似度最高的 k 个结果
  • 召回 (recall):相关结果中有多少被检索回来了
  • 精度 (precision):检索回来的结果里有多少是真相关的
  • Lost in the Middle:LLM 在长上下文中对中间位置内容关注度下降的现象
  • HyDE:用假答案的向量替代 query 向量去检索的技术
  • Reranker:对粗排结果重新打分排序的精排模型
  • Hybrid Search:同时用稠密 (语义) 和稀疏 (关键词) 检索并融合结果
  • Self-Query:LLM 自动从自然语言 query 中抽取结构化过滤条件
  • Graph RAG:基于知识图谱的检索增强生成

写在最后

RAG 是一个经验密集型的工程。同样一组策略,在不同数据上效果差异极大。所以最重要的事不是把所有策略学一遍,而是:

  1. 建好评测集——没有评测,所有改进都是猜
  2. 从简单方案做起——纯向量 → Hybrid → +Rerank,每加一步看指标真涨再保留
  3. 关注切块和检索这两个早期环节——它们的天花板决定全局上限
  4. 量力而行——Multi-Query、Self-Query、Graph RAG 都有合适和不合适的场景,别把”高级”等同于”好”

希望这份指南能让你少走一段弯路。也欢迎根据你自己的实战经验提 issue 或 PR。
